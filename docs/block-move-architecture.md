# 块移动方案分析

> 日期：2026-05-15
> 状态：方案 B（编辑器层面交换）已上线，方案 A（服务器 API）已验证可用

## 一、背景

块级浮动工具栏需要支持"上移/下移"和"拖拽移动"功能。核心问题是：如何在前端交换块的位置，并确保服务器数据一致。

## 二、两种方案

### 方案 A：服务器 API（`POST /blocks/:id/move`）

```
用户操作 → 调用 move API → 等待响应 → 重新加载编辑器内容
```

调用方式：
```json
POST /api/v1/blocks/:blockId/move
{
  "parentId": "b_root_xxx",
  "sortKey": "100000",
  "indent": 0,
  "createVersion": true
}
```

### 方案 B：编辑器层面交换（ProseMirror 事务）✅ 当前采用

```
用户操作 → ProseMirror 事务交换节点 → 触发 onChange → 防抖保存 → saveDocumentContent 位置 diff 同步
```

核心代码：
```typescript
tr.replaceWith(startA, endB, [nodeB, nodeA]);
view.dispatch(tr);
```

## 三、对比分析

| 维度 | 方案 A：服务器 API | 方案 B：编辑器交换 |
|------|-------------------|-------------------|
| **即时反馈** | ❌ 需等网络请求 + 重新加载 | ✅ 事务立即生效 |
| **数据一致性** | ✅ 服务器是 source of truth | ⚠️ 内容互换，sortKey 不变 |
| **实现复杂度** | 中（需处理异步 + 重载） | 低（复用现有管道） |
| **blockId 映射** | ❌ 编辑器无此映射 | ✅ 不需要 |
| **sortKey 控制** | ✅ 精确控制位置 | ❌ 无法直接影响 |
| **远距离移动** | ✅ 直接指定目标位置 | ⚠️ 需逐格交换或扩展事务 |
| **协作场景** | ✅ 服务器协调 | ⚠️ 可能冲突 |
| **用户体验** | ⚠️ 闪屏重载 | ✅ 丝滑无感知 |

### 方案 A 的优势

1. **精确控制位置**：通过 sortKey 可以将块移动到任意位置，不局限于相邻交换
2. **服务器权威**：移动操作直接在服务器完成，数据一致性有保障
3. **版本管理**：每次移动创建版本，有完整的操作历史

### 方案 A 的劣势

1. **需要 blockId 映射**：编辑器只知道位置，不知道 blockId。需要维护一个位置→ID 的映射表
2. **异步操作**：调用 API 后需要等待响应，然后重新加载编辑器内容，用户体验差
3. **实现复杂**：需要处理加载状态、错误重试、并发冲突等

### 方案 B 的优势

1. **即时反馈**：ProseMirror 事务立即生效，用户无感知
2. **利用现有管道**：onChange → 防抖保存 → saveDocumentContent 位置 diff，完全复用
3. **实现简单**：几行代码即可完成

### 方案 B 的劣势

1. **语义差异**：服务器端是"内容互换"而非"位置互换"。即 blockId 不变，payload 交换
2. **类型不匹配风险**：如果交换的两个块类型不同（如 heading 和 paragraph），可能出现类型-内容不匹配
3. **sortKey 不变**：无法精确控制位置，只能依赖位置 diff 的自动计算

## 五、可行性评估

### 当前方案（方案 B）的适用场景

- ✅ 相邻块交换（上移/下移）
- ✅ 单用户编辑
- ✅ 对 sortKey 精确度要求不高

### 需要切换到方案 A 的场景

- ❌ 远距离拖拽移动（跳过多个块）
- ❌ 多人协作编辑
- ❌ 需要精确控制块位置

### 混合方案（推荐）

短期使用方案 B（简单高效），长期在以下场景引入方案 A：

1. **拖拽移动**：当拖拽距离超过 2 个块时，调用服务器 API 精确定位
2. **协作场景**：多人编辑时，通过服务器 API 协调位置
3. **撤销/重做**：需要精确恢复历史位置时

## 六、后续建议

### 短期（当前）

- 方案 B 满足上移/下移需求，保持不变
- 菜单其他功能（删除、复制等）逐步接入

### 中期

- 实现拖拽移动时，考虑混合方案：
  - 相邻移动：方案 B（ProseMirror 事务）
  - 远距离移动：方案 A（服务器 API + 位置映射表）

### 长期

- 如果引入实时协作（Y.js / WebSocket），全面切换到方案 A
- 维护位置→blockId 映射表，支持精确位置控制

## 七、结论

**方案 B 是当前最优选择**：即时反馈、实现简单、用户体验好。服务器 API 作为备用方案，已验证可用，在未来需要精确位置控制或协作场景时可以引入。
